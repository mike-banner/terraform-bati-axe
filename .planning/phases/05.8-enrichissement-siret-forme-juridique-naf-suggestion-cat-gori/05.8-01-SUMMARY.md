# Phase 05.8 Plan 01: Fondations enrichissement SIRET Summary

Extension du parsing `lookupSiret()` pour exposer `legal_form`/`naf_code`, creation d'un mapping statique NAF -> categories BTP (`suggestCategoriesFromNaf`), et migration ajoutant deux colonnes nullable bornees sur `professionals`.

## Ce qui a ete fait

### Task 1 — `server/utils/siretLookup.ts`
`SiretLookupResult` expose desormais `legal_form?: string` et `naf_code?: string`, extraits de `entry.nature_juridique` et `entry.activite_principale` (deja presents dans la reponse de l'API recherche-entreprises.api.gouv.fr, aucun nouvel appel reseau). Peuples pour les statuts `active` et `closed` uniquement, absents pour `not_found`/`error`. `naf_code` normalise en majuscule + trim (point conserve, la normalisation sans point se fait cote mapping).

### Task 2 — `server/utils/nafCategoryMap.ts` (nouveau)
`NAF_TO_CATEGORIES` mappe 14 codes NAF batiment (divisions 41 construction + 43 travaux de construction specialises) vers les 6 valeurs exactes de `VALID_CATEGORIES` (`claim.post.ts`). `suggestCategoriesFromNaf(naf?)` normalise l'entree (upper, retire points/espaces) et resout via le mapping, `[]` sinon. Deux rattachements marques `// ponytail:` car ce sont des choix de jugement, pas des correspondances officielles INSEE : 4331Z (platrerie) -> isolation, 4399A (etancheite) -> toiture. Self-check integre au fichier (`_selfCheck()`, execute via `node server/utils/nafCategoryMap.ts` ou import direct).

### Task 3 — Migration `20260720020000_phase058_siret_enrichment.sql`
Deux colonnes nullable sur `public.professionals` : `siret_legal_form TEXT` (<=50 car.) et `siret_naf_code TEXT` (<=20 car.), via `ADD COLUMN IF NOT EXISTS` + pattern `CHECK ... NOT VALID` puis `VALIDATE CONSTRAINT` (reprend le style de la migration 05.7). Chaque CHECK tolere `NULL`.

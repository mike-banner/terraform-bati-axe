# Phase 5 — Plan 02 : Acquisition Pros (cold email)

> Objectif : recruter des artisans dans la base via email **gratuit** (Resend), en s'appuyant sur l'infra `prospects` déjà au schéma. Couvre REQ-05.
> **Source des prospects à trancher à l'exécution** (CSV / seed / scrape) — le plan reste agnostique sur l'origine de la donnée.

## Existant réutilisé (ne pas réinventer)
- Table `prospects` (`source`, `raw_data` JSONB, `company_name`, `siret` UNIQUE, `email`, `phone`, `zip_code`, `optin_status` enum `pending`, `optin_email_sent_at`, `converted_professional_id`) — migration `20260603000000_schema_initial.sql`.
- Table `consents` (`subject_type`, `subject_id`) pour la trace LCEN/RGPD.
- `server/api/v1/prospects/[id].get.ts` — sert le prospect (rejette si déjà converti).
- `server/api/v1/pro/claim.post.ts` — accepte `prospect_id`, doit marquer `converted_professional_id` au claim (à vérifier/compléter).
- `server/utils/email.ts` — `sendEmail({to,subject,html})` (mock en dev, Resend en prod).

## 1. Import des prospects
- **Fichier** : `server/api/v1/admin/prospects/import.post.ts` (admin only — check `ADMIN_EMAILS`, cf. pattern `admin/verify.post.ts`).
- **Action** : reçoit un tableau de prospects (Zod), insère dans `prospects` avec **upsert dédupliqué sur `siret`**, `source` renseignée, `optin_status='pending'`. Service Role pour contourner le RLS.
- **Note source** : à l'exécution, brancher soit un parsing CSV côté admin, soit un seed SQL, soit (plus tard, plan séparé) un scraper. Le endpoint reste la porte d'entrée unique.

## 2. Envoi des invitations (cold email)
- **Fichier** : `server/api/v1/admin/prospects/invite.post.ts` (admin only).
- **Action** : sélectionne les prospects `optin_status='pending'` ET `optin_email_sent_at IS NULL` ET `email IS NOT NULL`, en lot borné (ex. limit param). Pour chacun :
  - `sendEmail` avec lien `${baseUrl}/pro/claim?prospect=<id>`.
  - met à jour `optin_email_sent_at = now()`.
  - insère une ligne `consents` (`subject_type='prospect'`, `subject_id=<id>`, action='cold_email_sent').
  - **jamais** d'envoi si `optin_status='unsubscribed'`.
- **Garde-fou** : en dev les emails sont mockés (console) — tester le funnel sans quota. `EMAIL_LIVE=true` pour un vrai envoi de contrôle.

## 3. Pré-remplissage claim depuis l'invitation
- **Fichier** : `app/pages/pro/claim.vue` (existant).
- **Action** : lire `?prospect=<id>` dans l'URL, appeler `GET /api/v1/prospects/<id>`, pré-remplir `company_name` / `siret` / `postal_code`. Passer `prospect_id` au `POST /pro/claim`.
- **Fichier** : `server/api/v1/pro/claim.post.ts` — vérifier que `converted_professional_id` est bien positionné sur le prospect à la conversion (sinon l'ajouter).

## 4. Suivi funnel (acquisition)
- **Fichier** : `server/api/v1/admin/acquisition.get.ts` (admin only).
- **Action** : agrège les compteurs prospects par état — total, `pending`, invités (`optin_email_sent_at NOT NULL`), convertis (`converted_professional_id NOT NULL`), désinscrits. Renvoie taux de conversion.
- **Fichier** : `app/pages/admin/index.vue` — ajouter une carte « Acquisition » (chiffres bruts, design final plus tard).

## 5. Désinscription (RGPD/LCEN)
- **Fichier** : `server/api/v1/prospects/unsubscribe.get.ts` (public, par id/token).
- **Action** : passe `optin_status='unsubscribed'`, trace `consents`. Lien à inclure en pied de chaque cold email.

## Vérification (DoD)
- [ ] Import idempotent sur `siret` (re-import ne duplique pas).
- [ ] Invite n'envoie qu'aux `pending` non déjà invités, jamais aux `unsubscribed`.
- [ ] Email d'invitation → claim pré-rempli → `converted_professional_id` posé.
- [ ] `consents` tracé à l'envoi et à la désinscription.
- [ ] Funnel admin cohérent (somme des états = total).
- [ ] Tests unitaires sur la logique de sélection (qui reçoit / qui ne reçoit pas).

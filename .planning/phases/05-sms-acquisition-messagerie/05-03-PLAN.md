# Phase 5 — Plan 03 : Feedback loop lead + Email onboarding

> Objectif : fermer la boucle côté particulier (relancer la recherche) et poser les emails d'accueil pro derrière un flag off par défaut. Couvre REQ-06 et REQ-07.

## Existant réutilisé
- `app/pages/mon-projet/[token].vue` — Espace Client (magic link).
- `server/api/v1/magic-link/[token].get.ts` — sert projet + pros + messages.
- `server/api/v1/leads/[id]/claim.patch.ts` — porte le **cap concurrentiel (3 max)**.
- `server/api/v1/market-local.get.ts` — marché en pull (projets visibles selon le cap).
- `server/utils/email.ts` — `sendEmail`.

## Partie A — Feedback loop : refus → remise au marché auto (REQ-06)

### A0. État par lead
- **Fichier** : nouvelle migration `supabase/migrations/*_phase5_lead_feedback.sql`.
- **Action** : ajouter une colonne d'état particulier sur `leads`, ex. `customer_decision` ∈ {`pending`,`refused`,`selected`} défaut `pending` (ne pas confondre avec `lead_status` métier existant). Index sur `project_id` pour le check « tous refusés ».

### A1. Endpoint décision (refuser / retenir)
- **Fichier** : `server/api/v1/magic-link/[token]/decision.post.ts` (auth par `access_token` du projet, comme le reste de l'espace client).
- **Action** : reçoit `{ lead_id, decision: 'refused' | 'selected' }`, vérifie que le lead appartient bien au projet du token, met à jour `customer_decision`.
  - `selected` → **clôturer le projet** (sort du marché, plus de nouveaux pros).
  - `refused` → si **tous** les leads débloqués du projet sont désormais `refused` → **remise au marché automatique** (rendre le projet de nouveau visible dans `market-local` pour 3 nouveaux pros) + `sendEmail` au particulier (« votre projet est de nouveau visible »). **Conserver** leads + messages existants.
- **Garde-fou anti-spam** : borne/cooldown sur le nombre de remises au marché (ex. champ `relaunch_count` + `last_relaunched_at` sur `projects`).

### A2. UI espace particulier — décisions + profils pros
- **Fichier** : `app/pages/mon-projet/[token].vue`.
- **Action** : pour chaque artisan engagé, afficher une **carte** avec :
  - lien vers sa **fiche publique** (`/pro/[dept]/[slug]`) — voir Partie C / REQ-09 ;
  - boutons **« Pas intéressé »** (→ decision `refused`) et **« J'ai retenu celui-ci »** (→ decision `selected`), avec feedback visuel.
  - Pas de bouton « relancer » manuel : la remise au marché est automatique côté serveur quand tout est refusé. On affiche juste un message d'état (« nous cherchons de nouveaux artisans »).

## Partie C — Profil public pro depuis l'espace particulier (REQ-09)

### C1. Enrichir le retour magic-link
- **Fichier** : `server/api/v1/magic-link/[token].get.ts`.
- **Action** : pour chaque pro engagé sur le projet, renvoyer en plus de `company_name` : `canonical_slug` (ou `short_id`) + `category` + `logo_url` + `is_verified`, afin que le client construise le lien `/pro/[dept]/[slug]`. (Aujourd'hui seul `company_name` remonte via le join messages — ajouter la liste des pros du projet via `leads`→`professionals`.)

### C2. Fiche publique réutilisée
- **Existant** : `app/pages/pro/[dept]/[slug].vue` + `server/api/v1/pro/profile/[slug].get.ts` (lookup `short_id`). Rien à recoder — juste lier depuis l'espace particulier.
- **Invariant** : profil non vérifié → afficher avec `is_verified:false`, **jamais 404** (CLAUDE.md).

## Partie B — Email onboarding pro (REQ-07)

### B1. Flag d'activation
- **Fichier** : `.env.example` + `nuxt.config.ts` (`runtimeConfig`).
- **Action** : variable `ONBOARDING_EMAILS` (défaut `false`). Documenter dans `.env.example` (règle CLAUDE.md). Préfixe `NUXT_` côté Cloudflare pour le runtimeConfig.

### B2. Déclenchement
- **Fichier** : `server/api/v1/pro/claim.post.ts` (accueil au claim) et/ou `server/api/v1/admin/approve-pro.post.ts` (accueil à la validation).
- **Action** : si flag on → `sendEmail` de bienvenue (contenu texte simple, design plus tard). Si off → no-op. Email idempotent (ne pas renvoyer si déjà envoyé — marqueur type `onboarding_email_sent_at` sur `professionals`).

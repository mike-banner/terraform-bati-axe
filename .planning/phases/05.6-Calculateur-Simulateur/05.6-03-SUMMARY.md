# Phase 05.6 Plan 03: Calculateur Bento 6 étapes + Lead Wall Summary

- **Script** : état `form` étendu (`renovation_type`, `pieces[]`, `surface_m2`, `gamme` remplacent `category`/`description`/`budget_range`/`timeline_range`), `togglePiece` (multi-select push/splice), `isStepValid` à 6 branches (étape 2 auto-valide si `renovation_type === 'totale'`), `estimate = computed(() => computeEstimate(...))` jamais affiché hors révélation, `revealed = ref(false)`.
- **Soumission** : `handleSubmit` POST `/api/v1/projects` avec body `{ calculator_data: { renovation_type, pieces, surface_m2, gamme, estimate_min, estimate_max }, postal_code, customer_name, customer_email, customer_phone, cgu_accepted, sms_opt_in }` — `category`/`description`/`budget_range` ne sont plus envoyés (dérivés server-side par le plan 05.6-01). Sur succès : `revealed.value = true`.
- **Template étapes 1-5** : tuiles `.bento-card` charte 04.7, actif = `border-orange-500 bg-orange-50`, `min-h-11` (cible tactile 44px), grille `grid-cols-2 md:grid-cols-3` pour les pièces (icônes lucide `ChefHat`/`Bath`/`Sofa`/`BedDouble`/`Trees`/`LayoutGrid`), saut automatique de l'étape 2 (aller ET retour) si rénovation totale, input `range` natif pour la surface (rung 4 ladder — pas de lib slider), bannière d'éligibilité code postal 78955 conservée avec copie UI-SPEC mise à jour.
- **Lead Wall (étape 6)** : tuile fluide dans la même enveloppe `.bento-card` (pas de modal), titre/sous-texte verbatim UI-SPEC, champs nom/email/téléphone réutilisant `RE_EMAIL`/`RE_PHONE_FR`/`normalizePhone` existants, CGU obligatoire + SMS optionnel, CTA "Recevoir mon estimation gratuite".
- **Révélation** : bloc `v-if="revealed"` unique affichant la fourchette `estimate_min`–`estimate_max` (typo Display `clamp(2.5rem, 4vw + 1rem, 3.5rem)`, chiffre en `text-safety`), CTA `/mon-projet/${createdAccessToken}` + retour accueil. L'estimation n'apparaît nulle part ailleurs dans le template.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Regroupement de commits] Les 3 tasks livrées en un seul commit atomique**
- **Trouvé pendant** : planification de l'exécution.
- **Raison** : les 3 tasks du plan réécrivent le même fichier unique (`app/pages/simulateur.vue`) de façon cohésive — script complet (Task 1), puis étapes 1-5 du template (Task 2), puis Lead Wall + révélation + navigation (Task 3). Committer après chaque task nécessiterait soit un fichier syntaxiquement invalide entre les commits (template incomplet sans les étapes 6), soit un découpage artificiel du diff a posteriori sans valeur ajoutée.
- **Fichiers modifiés** : `app/pages/simulateur.vue`.
- **Commit** : `7fa6179`.

**2. [Rule 1 - Bug UX] Transition inter-étapes ajustée**
- **Trouvé pendant** : Task 2 (template).
- **Problème** : le plan suggérait de réutiliser directement `.reveal-item`/`reveal-up` en `<Transition>` pour l'animation entre étapes, mais ces keyframes sont conçues pour l'apparition au montage initial (staggered par `nth-child`), pas pour un cycle enter/leave répété à chaque changement de `v-if`/`v-else-if`.
- **Fix** : `<Transition name="fade" mode="out-in">` (CSS scoped, 0.2s) autour du bloc d'étapes pour la transition de contenu, tout en conservant `.reveal-item`/`.reveal` sur les tuiles internes de chaque étape pour l'apparition en cascade au montage — comportement conforme à la charte 04.7 sans réutilisation incorrecte des keyframes.
- **Fichiers modifiés** : `app/pages/simulateur.vue`.
- **Commit** : `7fa6179`.

# Phase 05.6 Plan 01: Persistance calculator_data + dérivation category Summary

Ajout de la colonne JSONB `calculator_data` sur `projects`, d'un util de dérivation pure `deriveTrades`/`derivePrimaryCategory` (mapping pièce→métiers, option-a), et extension de `POST /api/v1/projects` pour accepter les choix granulaires du calculateur, dériver la `category` server-side et composer description/budget_range quand absents — sans casser l'invariant du marché dynamique (Phase 4.6).

## Task 1 — Décision (résolue en amont, aucune écriture de code)

L'utilisateur a validé l'option **option-a** (dérivation statique pièce/type → métier primaire) avant l'exécution de ce plan, ainsi que la table de mapping pièce→métiers telle qu'écrite dans le plan (Task 2), sans amendement. Aucune alternative (option-b générique, option-c mini-étape UI) n'a été retenue. Cette décision conditionne la Task 2 (implémentation du mapping) et la Task 3 (branchement API).

## Task 2 — Migration + util de mapping (testé)

- `supabase/migrations/20260717000000_phase56_calculator_data.sql` : `ALTER TABLE public.projects ADD COLUMN IF NOT EXISTS calculator_data JSONB;`, additive, sans contrainte destructive, suivant le pattern verbatim de `20260624000000_phase5_siret_badges.sql`.
- `server/utils/calculatorMapping.ts` : fonction pure sans I/O, même forme que `qualifyScore.ts`.
  - `deriveTrades(renovation_type, pieces)` — union dédupliquée des métiers via la table statique pièce→métiers ; retourne les 6 métiers pour `renovation_type === 'totale'`.
  - `derivePrimaryCategory(renovation_type, pieces)` — métier dominant (comptage par occurrence sur les pièces sélectionnées), départagé par l'ordre de priorité fixe `['maconnerie','plomberie','electricite','peinture','isolation','toiture']` ; `'totale'` → `'maconnerie'`.
  - Table de mapping pièce→métiers exactement celle du plan (cuisine, salle_de_bain, salon, chambre, exterieur, autre).
- `tests/unit/calculator-mapping.test.ts` : 6 tests vitest couvrant `totale` (6 métiers), union dédupliquée sur une pièce, union sur pièces multiples (pas de doublon), bornage des 6 ids valides pour `derivePrimaryCategory`, cas `totale`, et tiebreak déterministe (`cuisine`+`salon` → `electricite`, car `electricite`/`peinture` sont à égalité et `electricite` précède dans l'ordre de priorité).

## Task 3 — Extension de l'API `/api/v1/projects`

- Schéma Zod : `calculator_data` requis (objet `{ renovation_type, pieces, surface_m2 (positive), gamme, estimate_min, estimate_max (non négatifs) }`) ; `category`, `description`, `budget_range` passés en `.optional()`. `cgu_accepted: z.literal(true)` conservé inchangé.
- Après validation, avant tout usage : `category = data.category ?? derivePrimaryCategory(...)`, `trades = deriveTrades(...)`, `description` et `budget_range` composés si absents (le second via un helper local `formatEuro` avec `toLocaleString('fr-FR')`, format `X € – Y €`).
- Ces valeurs dérivées alimentent `computeQualifyScore`, l'insert (`category`, `description`, `budget_range`, `calculator_data: { ...data.calculator_data, trades }`), le matching pro (`.contains('categories', [category])`) et les métadonnées `audit_logs` — remplaçant partout les anciennes lectures directes de `data.category`/`data.description`/`data.budget_range`.
- Reste inchangé verbatim : zones, fallback 78955, consents CGU/SMS, audit_logs, mock email, gestion d'erreurs.

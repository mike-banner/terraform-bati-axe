### Auto-fixed Issues

**1. [Rule 3 - Blocking] `supabase db push` bloqué par une collision de timestamp pré-existante**
- **Found during:** Task 3 (application de la migration locale)
- **Issue:** Deux fichiers de migration partagent le même timestamp `20260717000000` (`20260717000000_phase055_portfolio.sql` et `20260717000000_phase56_calculator_data.sql`), pré-existants avant ce plan. `supabase db push --local` échoue avec `duplicate key value violates unique constraint "schema_migrations_pkey"`. Ce problème est hors du périmètre de ce plan (fichiers non listés dans `files_modified`) — non corrigé, documenté ici pour visibilité.
- **Fix:** La nouvelle migration `20260719000000_phase57_input_limits.sql` a été appliquée directement à la DB locale via `psql "postgresql://postgres:postgres@127.0.0.1:54322/postgres" -f supabase/migrations/20260719000000_phase57_input_limits.sql`, sans passer par l'historique `schema_migrations` de Supabase CLI.
- **Vérification:** `SELECT conname, convalidated FROM pg_constraint WHERE conname LIKE '%_len';` retourne les 10 contraintes créées avec `convalidated = t` — succès sans erreur de données existantes hors-limite (critère d'acceptation de la Task 3 rempli).
- **Committed in:** aucun commit additionnel nécessaire (fichier SQL déjà committé en `81b0c41`).
- **Action restante pour l'utilisateur :** la collision de timestamp `20260717000000` doit être résolue séparément (renommer un des deux fichiers avec un timestamp unique) pour que `supabase db push`/`migration up` refonctionnent normalement en local et sur les environnements distants. Commande à lancer après correction : `supabase db push` (ou `supabase migration up --local`).

# Phase 05.7: Code Review Report

**Reviewed:** 2026-07-20T02:11:33Z
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

La phase durcit correctement la validation d'entrée sur le périmètre listé dans l'audit : les bornes `.max()` Zod ajoutées correspondent presque partout aux `maxlength` HTML côté client, et la migration SQL mire fidèlement les bornes serveur pour `professionals`, `projects`, `messages`, `completed_projects`. Les deux gaps découverts en cours de route (`phone` absent du schema `.strict()`, `policy_number` sans borne) ont bien été corrigés côté Zod.

Cependant, en corrigeant le bug `phone` absent du schema `.strict()` de `me.patch.ts`, le plan 02 a introduit une régression fonctionnelle réelle : le champ `phone` n'a jamais été ajouté aux `SELECT` de `me.get.ts` et `me.patch.ts`, donc bien que la sauvegarde réussisse en base, l'API ne renvoie jamais la valeur au client — le téléphone semble disparaître à chaque rechargement de `/espace/profil`. C'est un défaut bloquant, vérifiable en lisant les deux endpoints côte à côte.

Le reste des observations concerne des incohérences mineures dans l'application du pattern établi par la phase elle-même (mirror DB CHECK, `.trim()` systématique, bornage des tableaux) plutôt que des failles de sécurité critiques.

## Critical Issues

### CR-01: Le téléphone du profil pro ne peut jamais être ré-affiché après sauvegarde (SELECT incomplet)

**File:** `server/api/v1/pro/profile/me.patch.ts:48` et `server/api/v1/pro/profile/me.get.ts:11`
**Issue:** Le plan 02 a corrigé un bug réel (schema `.strict()` de `me.patch.ts` rejetait silencieusement `phone`) en ajoutant `phone` au `patchSchema` et à la mise à jour DB — mais **aucun des deux `SELECT`** (`me.get.ts` et le `SELECT` de relecture après `UPDATE` dans `me.patch.ts`) n'a été mis à jour pour inclure la colonne `phone`. Résultat vérifiable :
- `POST /espace/profil` → `saveProfile()` envoie `{ bio, categories, zone, phone }`, l'`UPDATE` réussit et persiste bien `phone` en base.
- Au chargement suivant, `useAsyncData('pro-profile-page', …)` appelle `GET /api/v1/pro/profile/me`, dont le `SELECT` ne contient pas `phone` → `data.profile.phone` est `undefined` → `profile.phone` reste à `''` dans `app/pages/espace/profil.vue:35`.
- L'utilisateur voit donc systématiquement un champ téléphone vide après rechargement, alors que la donnée est bien enregistrée côté serveur — régression silencieuse et trompeuse (l'utilisateur croit avoir perdu sa saisie et va la ressaisir en boucle).
**Fix:**
```ts
// me.get.ts et me.patch.ts : ajouter 'phone' au SELECT
const { data: pro, error } = await supabase
  .from('professionals')
  .select('id, canonical_slug, short_id, postal_code, categories, bio, zone, phone, logo_url, company_name, full_name, is_verified, subscription_status')
  .eq('id', user.id)
  .single()
```

## Warnings

### WR-01: Nouvelles bornes Zod (`phone`, `policy_number`) sans miroir CHECK en base

**File:** `server/api/v1/pro/profile/me.patch.ts:10`, `server/api/v1/pro/documents/upload.post.ts:8`, `supabase/migrations/20260719000000_phase57_input_limits.sql`
**Issue:** Le plan 01 établit explicitement le pattern « toute borne `.max()` serveur a son miroir `CHECK` en base, en dernier rempart » (voir en-tête de la migration). Le plan 02 a lui-même découvert et corrigé deux gaps de validation serveur (`phone` sur `professionals`, `policy_number` sur `verifications`) mais n'a ajouté **que** la contrainte Zod, sans migration SQL complémentaire pour ces deux colonnes. Le rempart base de données reste donc absent pour ces deux champs découverts dans cette même phase.
**Fix:** Ajouter une migration complémentaire :
```sql
ALTER TABLE professionals
  ADD CONSTRAINT professionals_phone_len CHECK (phone IS NULL OR char_length(phone) <= 20) NOT VALID;
ALTER TABLE professionals VALIDATE CONSTRAINT professionals_phone_len;

ALTER TABLE public.verifications
  ADD CONSTRAINT verifications_policy_number_len CHECK (policy_number IS NULL OR char_length(policy_number) <= 50) NOT VALID;
ALTER TABLE public.verifications VALIDATE CONSTRAINT verifications_policy_number_len;
```

### WR-02: Tableaux non bornés sur l'endpoint public `POST /api/v1/projects`

**File:** `server/api/v1/projects.post.ts:20-21`
**Issue:** `calculator_data.pieces: z.array(z.string().max(100))` n'a pas de `.max()` sur la longueur du tableau lui-même. Cet endpoint est **public, non authentifié** (le simulateur est accessible sans compte). Un appelant direct de l'API (hors UI, qui limite `pieces` à 6 valeurs fixes) peut soumettre un tableau de plusieurs milliers d'entrées de 100 caractères chacune, gonflant artificiellement la `description` générée server-side (`pieces.join(', ')`, ligne 60) et la taille de la ligne `projects` insérée. Même remarque pour `categories` dans `claim.post.ts:18` (`z.array(z.enum(...)).min(1)` sans `.max()`), bien que le risque y soit moindre car authentifié.
**Fix:**
```ts
pieces: z.array(z.string().max(100)).max(10),
```
et dans `claim.post.ts` :
```ts
categories: z.array(z.enum(VALID_CATEGORIES)).min(1).max(6, 'Sélectionnez au maximum 6 corps de métier.'),
```

### WR-03: `image_urls` sans borne haute dans `realisations/index.post.ts`

**File:** `server/api/v1/pro/realisations/index.post.ts:9`
**Issue:** `image_urls: z.array(z.string().url()).min(1)` n'a pas de plafond. Un pro authentifié peut soumettre un tableau arbitrairement long d'URLs dans une seule requête `POST`, sans limite alignée avec l'usage réel (une réalisation affiche typiquement quelques photos).
**Fix:**
```ts
image_urls: z.array(z.string().url('URL de photo invalide.')).min(1, 'Au moins une photo est requise.').max(10, 'Maximum 10 photos par réalisation.'),
```

### WR-04: `.trim()` appliqué de façon incohérente entre les formulaires de cette phase

**File:** `app/components/RealisationForm.vue:104`, `app/pages/espace/leads/[id].vue:78`
**Issue:** Le plan 02 documente explicitement l'ajout de `.trim()` avant envoi comme pattern de la phase, mais ne l'applique qu'à `simulateur.vue` et `claim.vue`. `RealisationForm.vue` envoie `title: title.value` (non trim) alors que la garde de validation locale (`title.value.trim()`) vérifie une version trimée — un titre du type `"  Toiture  "` passe la validation mais est stocké avec les espaces. Idem pour `content: chatInput.value` dans `leads/[id].vue`, qui envoie la valeur brute malgré une garde `chatInput.value.trim()` à l'appel.
**Fix:** Trim au moment de la construction du payload, cohérent avec le reste de la phase :
```ts
// RealisationForm.vue
body: { title: title.value.trim(), description: description.value.trim() || null, city: city.value.trim() || null, image_urls }
// leads/[id].vue
body: { lead_id: claimId.value, content: chatInput.value.trim() }
```

### IN-01: Formulaire `profil.vue` sans retour d'erreur inline pour le téléphone

**File:** `app/pages/espace/profil.vue:128-136`
**Issue:** Contrairement aux autres formulaires de la phase (`simulateur.vue`, `pro/claim.vue`), le champ `phone` de `profil.vue` a un `pattern` HTML mais aucun état `touched`/erreur associé, et le `<form>` n'a pas `novalidate` — la validation native du navigateur bloquera silencieusement la soumission avec un message non stylé, incohérent avec le reste de l'UI (bordure rouge + message sous les autres champs de la phase).
**Fix:** Ajouter un état d'erreur inline cohérent avec le pattern déjà utilisé ailleurs dans la phase (ex. `contactErrors` de `simulateur.vue`), ou a minima documenter que c'est un choix assumé.

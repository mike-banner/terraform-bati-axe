# Phase 5.5 Plan 03: API CRUD réalisations pro Summary

Trois endpoints Nitro légers (liste, création, suppression) pour que le pro authentifié gère ses réalisations `completed_projects`, entièrement scopés par la policy RLS `manage_own_completed_projects` — aucune vérification d'ownership applicative n'est nécessaire.

## Ce qui a été fait

- **`index.get.ts`** : auth via `serverSupabaseClient`, `select('*').eq('professional_id', user.id).order('created_at', { ascending: false })`, retour `{ status: 'SUCCESS', realisations }`.
- **`index.post.ts`** : Zod `.strict()` (title requis, description ≤500, city optionnelle, image_urls tableau d'URLs non vide). Insert avec `professional_id: user.id`. Le champ de mise en avant (réservé admin) est absent du schéma de validation, donc rejeté silencieusement par `.strict()` si envoyé.
- **`[id].delete.ts`** : `getRouterParam(event, 'id')` validé comme UUID via Zod, `delete().eq('id', id)` — le RLS empêche toute suppression hors périmètre du pro connecté (0 ligne affectée si non-propriétaire, pas d'erreur explicite mais pas de fuite non plus).

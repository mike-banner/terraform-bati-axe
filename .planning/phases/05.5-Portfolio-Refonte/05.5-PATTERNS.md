## Pattern Assignments

### `server/api/v1/pro/profile/logo-presign.post.ts` (route, file-I/O) — BUG FIX

**Analog:** `server/api/v1/pro/documents/presign.post.ts` (already correct)

**The bug** (current file, lines 55-62):
```typescript
// 5 MB max (T-04.5-16) — encoded via Content-Length in signed headers
const MAX_SIZE = 5 * 1024 * 1024 // 5_242_880

const request = await aws.sign(url, {
  method: 'PUT',
  headers: { 'Content-Type': content_type, 'Content-Length': String(MAX_SIZE) },
  aws: { signQuery: true }
})
```
`Content-Length` is forced to the theoretical max, but the browser's `fetch`/`XHR` PUT sends the *real* file's `Content-Length` → AWS SigV4 signature mismatch (`SignatureDoesNotMatch`).

**Correct pattern** (`documents/presign.post.ts` lines 73-77):
```typescript
// On ne passe PAS de headers ici pour éviter que la signature ne soit trop stricte.
const request = await aws.sign(url, {
  method: 'PUT',
  aws: { signQuery: true }
})
```
**Fix:** remove the `headers` object entirely from `aws.sign()` in `logo-presign.post.ts`; keep the 5 MB check as a client-side/Zod guard only (already enforced client-side in `espace/profil.vue` `MAX_SIZE` check). Apply the exact same no-headers pattern to the new multi-file `pro/realisations/presign.post.ts` route — do not repeat the bug.

**Idempotent policy pattern** (lines 16-35):
```sql

DO $$
BEGIN
    IF NOT EXISTS (
        SELECT 1 FROM pg_policies WHERE tablename = 'messages' AND policyname = 'Pros can view messages for their leads'
    ) THEN
        CREATE POLICY "Pros can view messages for their leads"
        ON public.messages
        FOR SELECT
        USING (
            EXISTS (
                SELECT 1 FROM public.leads l
                WHERE l.id = messages.lead_id
                AND l.pro_id = auth.uid()
            )
        );
    END IF;
END $$;
```

### `server/api/v1/admin/realisations-showcase.post.ts` (route, CRUD toggle)

**Analog:** `server/api/v1/admin/approve-pro.post.ts` (full file read above) — closest exact match for "admin toggles a boolean flag on a row owned by a pro."

**Full pattern to copy (imports, auth, validation, update, audit log)** (lines 1-21, 57-77):
```typescript
import { z } from 'zod'
import { serverSupabaseUser, serverSupabaseServiceRole } from '#supabase/server'

const schema = z.object({
  project_id: z.string().uuid(),
  is_showcased: z.boolean()
})

export default defineEventHandler(async (event) => {
  const user = await serverSupabaseUser(event)
  if (!user) throw createError({ statusCode: 401, statusMessage: 'Non autorisé.' })

  const isAdmin = (user as any).app_metadata?.role === 'admin'
  if (!isAdmin) throw createError({ statusCode: 403, statusMessage: 'Accès réservé aux administrateurs.' })

  const body = await readBody(event)
  const parsed = schema.safeParse(body)
  if (!parsed.success) throw createError({ statusCode: 400, statusMessage: 'Données invalides.' })

  const { project_id, is_showcased } = parsed.data
  const supabase = await serverSupabaseServiceRole(event) as any

  await supabase.from('completed_projects').update({ is_showcased }).eq('id', project_id)

  await supabase.from('audit_logs').insert({
    actor_id: (user as any).id,
    action: 'showcase_toggled',
    target_table: 'completed_projects',
    target_id: project_id,
    metadata: { is_showcased }
  })

  return { status: 'SUCCESS', is_showcased }
})
```

### `server/api/v1/projects/[id]/like.post.ts` (route, event-driven)

**Analog (validation + insert + graceful duplicate handling):** `server/api/v1/admin/verify.post.ts` for the try/catch + Zod shape; unique-constraint approach mirrors `free_lead_grants` (`20260609000000_phase45_schema.sql` line 17 `CONSTRAINT unique_pro_lead_grant UNIQUE (pro_id, lead_id)`).

**Pattern:**
```typescript
import { z } from 'zod'
import { serverSupabaseServiceRole } from '#supabase/server'
import { createHash } from 'node:crypto'

const schema = z.object({ project_id: z.string().uuid() })

export default defineEventHandler(async (event) => {
  const body = await readBody(event)
  const parsed = schema.safeParse(body)
  if (!parsed.success) throw createError({ statusCode: 400, statusMessage: 'Requête invalide.' })

  const ip = getRequestIP(event, { xForwardedFor: true }) || 'unknown'
  const ipHash = createHash('sha256').update(`${ip}:${parsed.data.project_id}`).digest('hex')

  const supabase = await serverSupabaseServiceRole(event) as any
  const { error } = await supabase.from('likes').insert({ project_id: parsed.data.project_id, ip_hash: ipHash })

  // Duplicate = unique constraint violation (Postgres code 23505) → already liked, not an error
  if (error && error.code !== '23505') {
    throw createError({ statusCode: 500, statusMessage: error.message })
  }

  const { count } = await supabase
    .from('likes')
    .select('*', { count: 'exact', head: true })
    .eq('project_id', parsed.data.project_id)

  return { status: 'SUCCESS', likes: count ?? 0 }
})
```
No `try/catch` wrapper needed here per Ponytail — Nitro's `defineEventHandler` propagates thrown `createError` natively, matching the plainer style of `queue.get.ts` (no top-level try/catch) rather than `verify.post.ts`'s heavier wrapper.

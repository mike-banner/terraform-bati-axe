# Phase 05.5: Portfolio-Refonte - Research

## 1. Domain Investigation

**A. Bug Upload Logo R2**
- **Cause identifiée** : La route de pre-signature `server/api/v1/pro/profile/logo-presign.post.ts` inclut des `headers` lors de l'appel à `aws.sign()` (notamment `Content-Length: String(MAX_SIZE)`). Cela force la signature AWS v4 à attendre *exactement* un header `Content-Length` de 5 Mo lors de la requête `PUT` du navigateur. Cependant, `fetch()` dans le navigateur calcule son propre `Content-Length` en fonction du fichier réel, ce qui provoque une erreur `SignatureDoesNotMatch` côté Cloudflare R2.
- **Solution** : Retirer l'objet `headers` lors de l'appel `aws.sign(url, { method: 'PUT', aws: { signQuery: true } })`, exactement comme c'est déjà implémenté correctement dans `server/api/v1/pro/documents/presign.post.ts`.

**B. Modèle de Données (Portfolio & Social Proof)**
- La table `completed_projects` nécessitera une migration SQL. Elle doit être liée à la table `professionals` via une Foreign Key.
- Champs requis : `id`, `professional_id`, `title`, `description`, `city`, `image_urls` (TEXT[]), `is_showcased`, `created_at`.
- La table `likes` servira à stocker les clics d'appréciation sur les projets avec un système anti-spam basique. Un IP hash ou un cookie (stocké en DB ou géré côté Edge) fera l'affaire.

**C. Refonte Profil Public Immersif**
- Le fichier `app/pages/pro/[dept]/[slug].vue` utilise actuellement le layout par défaut. Pour le rendre immersif, il faudra appliquer `definePageMeta({ layout: false })` (ou créer un layout `immersive.vue`).
- Navigation minimale exigée par le PRD : un bouton flottant ou un fil d'ariane pour revenir à l'accueil, en éliminant la navbar générique.
- Design : application stricte du guide `ui-ux-pro-max` (Bento grids, Mobile First absolu).

**D. Landing Page & Carousel**
- Le fichier `app/pages/index.vue` contient déjà une structure Bento Grid (variante C). Il faudra y insérer une nouvelle section affichant les `completed_projects` ayant `is_showcased = true`.
- Puisque l'application utilise SSR (Nuxt 3), le carousel de réalisations devra être implémenté via des composants CSS-only ou des librairies légères compatibles SSR (ex: Embla Carousel Vue ou CSS scroll-snap) pour ne pas pénaliser le CLS / SEO.

## 2. Technical Decisions & Assumptions
1. **Migration SQL** : Un script `2026xxxx_phase055_portfolio.sql` devra être écrit et intégré via le système de migration Supabase habituel.
2. **R2 Multi-upload** : Le backoffice pro (`app/pages/espace/profil.vue` ou une modale/page dédiée) devra gérer un upload multiple asynchrone en boucle (un presign par image), en s'assurant que le correctif du `aws.sign()` soit bien appliqué globalement.
3. **Sécurité RLS** : 
   - `completed_projects` : `SELECT` public, `INSERT/UPDATE/DELETE` restreint au `professional_id` authentifié. L'admin pourra éditer `is_showcased` via Service Role.
   - `likes` : `INSERT` ouvert (RPC ou Endpoint sécurisé avec rate limit pour l'anti-spam).

## 3. Validation Architecture
- **API Tests** : Vérifier que `logo-presign.post.ts` génère bien une URL sans forcer un `Content-Length` strict.
- **Database** : Vérifier que les tables `completed_projects` et `likes` ont bien des règles RLS définies.
- **UI Tests** : Le profil public ne doit pas contenir de balise `<header>` provenant du layout par défaut.

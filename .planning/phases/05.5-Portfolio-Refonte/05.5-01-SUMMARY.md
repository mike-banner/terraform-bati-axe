# Phase 05.5 Plan 01: Schéma DB Portfolio & Likes Summary

Migration Supabase créant `completed_projects` (galerie de réalisations pro) et `likes` (compteur anti-spam avec contrainte anti-doublon), avec policies RLS conformes au SPEC.

## What Was Built

- `public.completed_projects` : 8 colonnes exactes du SPEC (`id`, `professional_id`, `title`, `description`, `city`, `image_urls`, `is_showcased`, `created_at`), FK `professional_id → public.professionals(id) ON DELETE CASCADE`.
- Policies RLS sur `completed_projects` : `select_public_completed_projects` (SELECT public, `USING (true)`) et `manage_own_completed_projects` (ALL authenticated, `auth.uid() = professional_id`). Pas de policy pour `is_showcased` — le toggle admin passera par service_role (plan 04).
- `public.likes` : `id`, `project_id → public.completed_projects(id) ON DELETE CASCADE`, `ip_hash`, `created_at`, `CONSTRAINT unique_project_ip_like UNIQUE (project_id, ip_hash)`. Index `idx_likes_project_id`.
- RLS activé sur `likes` sans aucune policy — accès refusé à `public`/`authenticated`, seuls les endpoints `service_role` (plan 04) pourront écrire/compter les likes. `ip_hash` est un texte opaque (SHA-256 calculé côté endpoint), jamais l'IP brute.

**Bloqué par l'environnement d'exécution.** Ce worktree agent tourne dans un sandbox WSL sans accès au daemon Docker (`Cannot connect to the Docker daemon at unix:///var/run/docker.sock`, pas de `systemd`, pas de `sudo` disponible sans mot de passe). La stack Supabase locale (port 54321/54322) n'est pas démarrée et ne peut pas l'être depuis cet agent :

```
failed to inspect container health: Cannot connect to the Docker daemon at unix:///var/run/docker.sock. Is the docker daemon running?
```

Ce n'est pas un bug de la migration ni un problème auto-réparable (Rules 1-3 ne s'appliquent pas — ce n'est ni un bug de code, ni une fonctionnalité manquante, ni une install de package). C'est un gate d'environnement, comparable à un auth gate : nécessite Docker Desktop démarré côté hôte Windows avec l'intégration WSL active pour cette distro.

**Action requise avant merge / prochaine vague :** dans une session avec Docker Desktop actif, lancer :
```bash
supabase db reset
psql "postgresql://postgres:postgres@127.0.0.1:54322/postgres" -tAc "select count(*) from information_schema.tables where table_schema='public' and table_name in ('completed_projects','likes')"
psql "postgresql://postgres:postgres@127.0.0.1:54322/postgres" -tAc "select conname from pg_constraint where conname='unique_project_ip_like'"
psql "postgresql://postgres:postgres@127.0.0.1:54322/postgres" -tAc "select count(*) from pg_policies where tablename='completed_projects'"
```
Le fichier SQL a été relu manuellement contre le SPEC (colonnes, FK, policies, contrainte unique) — aucune erreur de syntaxe attendue, mais l'application effective en local reste à confirmer par le développeur ou l'orchestrateur dans un environnement avec Docker.

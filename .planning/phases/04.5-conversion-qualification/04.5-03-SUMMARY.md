# Plan 04.5-03 Summary — Free-Lead Logic + maskLead + qualify_score

**Status:** Complete
**Wave:** 3
**Commit:** 3871c2d

## What was built

### server/utils/maskLead.ts
Added `isFreeGranted: boolean = false` as 4th parameter. Unlock condition extended to:
```
isUnlocked = isPremium || isFreeGranted || (unlocked_at !== null && unlocked_at <= now)
```
Default `false` preserves all existing call sites. D-03 comment added.

### server/api/v1/leads/index.get.ts
After resolving pro, fetches all `free_lead_grants` rows for the calling pro and builds a `Set<string>`. Passes `grantedSet.has(lead.id)` as `isFreeGranted` to maskLead. List endpoint is read-only — no counter mutation.

### server/api/v1/leads/[id].get.ts
Added `free_leads_used` to pro SELECT. On detail view by a Basic pro with no time-unlock:
1. Checks for existing grant (idempotent re-view)
2. If none and `free_leads_used < 3`: INSERT into `free_lead_grants`; on success or 23505 collision, increments counter with optimistic concurrency (`.eq('free_leads_used', current)`)
3. If `free_leads_used >= 3`: no grant, lead stays masked

Premium pros and time-unlocked leads skip grant logic entirely (D-04).

### server/api/v1/projects.post.ts
Computes 4 criteria before insert (D-11):
- `qualify_budget`: budget_range non-empty
- `qualify_phone`: customer_phone non-empty
- `qualify_description`: description.length > 50 (strict greater-than)
- `qualify_returning`: customer_email exists in prior projects row (SELECT with count before INSERT)

`qualify_score` = count of true criteria (0–4). All 5 columns added to INSERT payload and audit_log metadata.

## Design decisions
- `qualify_returning` lookup happens before INSERT — avoids counting the current submission
- UNIQUE(pro_id, lead_id) + 23505 guard handles concurrent first-views without double-granting
- Optimistic concurrency on counter (WHERE free_leads_used = N) prevents double-increment

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated pro → PATCH /me | Service-role UPDATE bound to auth.uid() — pro cannot modify another pro. |
| Authenticated pro → POST /logo-presign | Presigned URL scoped to a pro-specific key (`pro-logos/${user.id}/...`); even if URL leaks, scope is limited to one path. |
| Anonymous → GET /[slug] | Public read; new fields are intentionally public (D-16). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04.5-15 | Tampering | Pro forges canonical_slug change to take over another slug | mitigate | Zod .strict() rejects unknown keys; even if the key were forwarded, RLS + service-role .eq('id', user.id) ensures only the pro's own row is touched. Slug uniqueness constraint at DB further prevents collisions. |
| T-04.5-16 | Tampering | Pro uploads 1 GB file via presigned URL | mitigate | 5 MB ContentLength condition enforced in presign policy. Stripe-style hard cap. |
| T-04.5-17 | Tampering | Pro uploads executable disguised as image (e.g., .svg with embedded JS) | mitigate | SVG IS in the allow-list per D-17, so XSS via embedded JS is a real risk. Mitigation: serve all `pro-logos/*` keys with `Content-Disposition: attachment` OR with a `Content-Security-Policy: sandbox` header at the R2-public-URL boundary. If R2 does not support per-object CSP, add a Nuxt server middleware that proxies logo URLs and strips inline scripts from SVGs (defer to a follow-up if non-trivial; record as known risk). |
| T-04.5-18 | Information Disclosure | Public profile GET exposes bio/zone that pro intended internal | accept | D-16 explicitly states changes are immediate and publicly visible — that is the feature. |
| T-04.5-19 | Elevation of Privilege | Pro PATCHes subscription_status to active | mitigate | .strict() Zod schema does not list subscription_status as an allowed key; rejected before reaching DB. |
| T-04.5-SC | Tampering | No new npm packages (reuses existing aws-sdk client) | accept | No installs. |
</threat_model>

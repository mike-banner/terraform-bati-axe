<objective>
Implement the server-side logic for the 3-free-leads verrou (CNV-01) and the auto-computed qualification score (CNV-03):

1. Extend `maskLead` to accept a fourth boolean `isFreeGranted` and treat it as equivalent to `isPremium || unlocked_at <= now` for the unlock decision.
2. Extend `GET /api/v1/leads` to fetch the set of free-granted lead IDs for the calling pro and pass the resolved boolean per lead.
3. Extend `GET /api/v1/leads/[id]` to atomically issue a new grant on first detail view when the pro is Basic AND `free_leads_used < 3` AND the lead is not already granted AND the lead is not already 72h-unlocked. The grant inserts a row into `free_lead_grants` AND increments `professionals.free_leads_used` AND must not double-count if the same pro views the same lead twice (use `INSERT ... ON CONFLICT (pro_id, lead_id) DO NOTHING RETURNING id` and increment only when a row is returned).
4. Extend `POST /api/v1/projects` to compute `qualify_score` + 4 criterion booleans at insert time based on D-11 rules. The "returning client" criterion is satisfied when the `customer_email` value matches at least one OTHER existing row in `projects` (not the row being inserted).

Per D-03, "first 3" identification uses the `free_lead_grants` junction table — explicit & queryable rather than chronological-rank computation. This avoids race conditions when multiple lead views land concurrently.

Purpose: Core monetization logic. Drives the paywall trigger.
Output: All four files updated, masking covers all four code paths, qualify_score persisted on every new project.
</objective>
